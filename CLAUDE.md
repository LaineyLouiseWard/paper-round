# Paper Round

Daily literature monitor: screen.py fetches and dedups journal feeds, a
cloud routine screens new papers against the user's written scope, emails
a digest, and files keepers into Zotero. You are here to set it up or
tune it.

## Files

- screen.py — the pipeline. No user settings inside; do not edit it
- config.yaml — all settings (email, Zotero, sources). Edit this instead
- .env — keys, copied from .env.example. Gitignored; never commit keys
- source_list.md — the journals. Parsed by regex: entries must stay
  exactly `### N. Name` + `` - **Feed:** `url` ``; the rest is free-form
- research_scope.md + relevance_rules.md — the screening prompt
- routine_prompt.md — the cloud routine's task. The integrity rules at
  its top are load-bearing; never weaken them
- paper_log.csv, seen_papers.txt — generated by runs; don't hand-edit

## Rules

- Adding a journal the user names: try its publisher's RSS pattern
  (table in source_list.md). If the dry-run shows 0 entries, look up the
  ISSN yourself (https://api.crossref.org/journals?query=NAME) and use a
  fallback in config.yaml: AMS → crossref_fallbacks; Elsevier, Nature,
  or anything else → openalex_sources (needs OPENALEX_API_KEY in .env).
  A journal counts as added only when the dry-run shows entries from it.
- Papers in digests and logs come verbatim from fetched feed data.
  Never let an edit soften this.
- After any source or config change: python screen.py --dry-run
  (free, no keys needed).

## Setup

Follow the Start-here prompt in README.md: interview one topic at a
time, draft the scope from an abstract AND a Zotero CSV export, keys
from the README table, dry-run, then the cloud steps one confirmation
at a time.

## After setup

- "This wasn't relevant" → add an exclusion or narrow a trigger in
  relevance_rules.md; say what you changed
- "It missed a paper" → new trigger; check the journal is monitored
- "What did it find on X?" → answer from paper_log.csv, never memory
- Commit and push your changes so tomorrow's run uses them
